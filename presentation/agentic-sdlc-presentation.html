Context

Context

You're already in the Agentic SDLC era — AI is part of how you build software.
The question is how you use it.

The era

Agentic SDLC

AI agents participate across the entire development lifecycle — requirements, design, planning, implementation, testing, documentation.

Unstructured

Vibe Coding

Prompt → code → accept/reject → repeat.
No plan, no artifacts, no shared context. Fast for prototypes — breaks at team scale.

Structured

Spec-Driven Development (SDD)

Co-architect the plan, iterate until it's right, then delegate the execution. Decisions become artifacts — reviewable, resumable, diffable.

The frame for everything that follows
Unstructured

What is vibe coding?

You describe what you want in natural language. AI generates the code. You accept, reject, or iterate. No spec, no plan, no structure — just vibes.

Works great for
  • Quick prototypes and proof-of-concepts
  • One-off scripts and automation
  • Solo hacking and throwaway experiments
  • Learning a new framework or API
Breaks down at
  • Team scale — your AI context ≠ my AI context
  • Multi-session features — decisions evaporate overnight
  • Code review — giant diffs with no rationale
  • Reproducibility — same prompt, different results tomorrow

Vibe coding isn't wrong — it's just incomplete. The question is what comes next when the vibes stop scaling.

The starting point
The shift

Stop prompting. Start specifying.

Writing code isn't the job anymore. Deciding what to build and how it fits is. Think like a principal — own the design, the tradeoffs, the architecture. Let AI handle the mechanics.

Before · IC mindset

"Write the code"

Hands on keyboard. Syntax, loops, glue. AI as autocomplete. Design lives in your head — or nowhere.

  • Measured by lines shipped
  • Low-level decisions dominate the day
  • Architecture is a side effect
→
After · Principal mindset

"Design the system"

Own intent, boundaries, contracts, tradeoffs. AI drafts the implementation against a spec you already agreed on.

  • Measured by decisions that hold up
  • High-level design is the deliverable
  • Code is the output, not the input
The mental model

You move up a level. Architect, reviewer, decider. AI is the junior engineer: brilliant, fast, forgets overnight.
Feed it the artifacts a human teammate would need: use cases, design, plan.

The pitch
Structured

What is Spec-Driven Development?

The name says it: you drive with specs. Write them, polish them, then let AI execute against them.

Write

Specs

Intent, design, plan — captured as artifacts in the repo. Not in your head, not in a chat.

→
Polish

Iterate until sharp

Review with AI. Review with humans. Tighten the brief before anyone writes code.

→
Execute

AI ships it

Code gets written against a spec you already approved. You review the output, not guess the intent.

Why it works

Specs are diffable, reviewable, resumable. Decisions stop evaporating between sessions. Teammates — and future-you — can pick up the same context.

The destination
In the repo

What it looks like for a dev

Specs are real files in the repo. You work them first. Code shows up later — as the output of specs that are already sharp.

Phase 1 · You

Work the specs. Polish.

my-app/
├── specs/                ← gitignored, local only
│   ├── use-cases/        ← what we're building
│   │   ├── auth.md
│   │   └── checkout.md
│   ├── design/           ← how it fits
│   │   └── architecture.md
│   └── plans/            ← steps to ship
│       └── checkout-v1/
│           ├── README.md
│           └── step-01.md
├── src/                  ← untouched
├── .gitignore            ← specs/ listed here
└── CLAUDE.md             ← rules of the house

You iterate here. No code written yet.

→
execute
Phase 2 · AI

Code drops out of the specs.

my-app/
├── specs/                ← gitignored, your local brief
│   ├── use-cases/
│   ├── design/
│   └── plans/checkout-v1/
├── src/                  ← the only thing that ships
│   ├── auth/                  ← from plans/auth-v1/
│   │   ├── session.ts
│   │   └── session.test.ts
│   ├── checkout/              ← from plans/checkout-v1/
│   │   ├── cart.ts
│   │   ├── payment.ts
│   │   └── __tests__/
│   └── shared/
├── .gitignore
└── CLAUDE.md

Code is an output. Specs stay the input.

$ git log --oneline
a1b4e9f feat(checkout): implement payment flow
7d2cc18 feat(checkout): add cart service
5e22b41 feat(auth): add session handling
// specs/ stays local — only code lands in git

Specs are your working brief. Only code ships to the repo history.

Close the loop · extra step

Documenting

Extra step — not automatic.
One brief per domain, written from shipped code.
Registered in CLAUDE.md with a "read me when…" hint — Claude reads it when the topic comes up.

Phase 2 · shipped

Code, no docs.

my-app/
├── specs/                ← local, your brief
├── src/
│   ├── auth/
│   ├── checkout/
│   └── shared/
├── docs/                 ← missing — no context for teammates
├── .gitignore
└── CLAUDE.md

Reviewers see code with no committed rationale.

→
document
Phase 3 · AI

Docs aligned to code.

my-app/
├── specs/                ← unchanged, local
├── src/
│   ├── auth/
│   ├── checkout/
│   └── shared/
├── docs/                 ← one brief per domain
│   ├── auth.md
│   ├── checkout.md
│   └── architecture.md
├── .gitignore
└── CLAUDE.md             ← registers each doc + when to read it

CLAUDE.md = the index. Agents load only what they need.

$ git log --oneline
c4f2a11 docs: align with shipped checkout + auth
a1b4e9f feat(checkout): implement payment flow
7d2cc18 feat(checkout): add cart service
// docs ship to repo. specs never do.

Specs = private brief (local). Docs = public contract, pre-loaded context for every agent session.

Team-ready, not solo-ready
