Terminology

Mental model: skills, plugins, marketplaces

  • Skill — packaged instruction set (name + description + SKILL.md).
    Two invocation modes:
    • Auto-invoked — Claude matches your request against the description and loads the skill automatically.
    • Explicit — you type /skill-name in the prompt to force-invoke.
  • Plugin — one or more skills bundled with a plugin.json.
  • Marketplace — a Git repo that lists plugins. Anyone can publish one.

How they nest

marketplace (forge)
└── plugin (spec-driven-dev)
    ├── skill (use-cases-generator) → /use-cases-generator
    ├── skill (tech-design-generator) → /tech-design-generator
    ├── skill (dev-plan) → /dev-plan
    └── ... 6 more

Skills are discoverable — Claude picks the right one from descriptions. Or invoke explicitly with the slash command.

Install

Getting the marketplace and the plugin

1 · Add the marketplace

/plugin marketplace add https://github.com/dorg-atsabari/forge

Registers forge as a source. One-time per machine.

2 · Install the plugin

/plugin install spec-driven-dev@forge

Pulls the plugin's 9 skills into Claude Code.

What you get after install
  • 9 skills available in any project
  • Artifacts land in specs/ at the repo root
  • Just markdown — nothing runs in the background
Two commands. That's it.

Skill 01 / 09

use-cases-generator

/use-cases-generator

Turn raw requirements into a structured set of use cases organized by product surface.

In

A requirements doc, a Slack thread, a meeting transcript, or just a description.

Out

specs/use-cases-<system>/ — README plus per-use-case markdown grouped by interaction boundary, not functional area.

When to invoke: at the very start of a new feature, when requirements are messy or scattered.
Notable: two-level hierarchy — use cases (surfaces) contain flows (concrete journeys). Asks for approval of the proposed structure before writing files.

Skill 02 / 09

tech-design-generator

/tech-design-generator

Translate product requirements into a structured technical architecture: components, data stores, contracts, flows.

In

A spec, PRD, or use-case set — plus the existing codebase as context.

Out

specs/tech-design-<system>/ with design.md, dev-setup.md, api-contract.md, shared-patterns.md, and per-flow files.

When to invoke: real architecture decisions. Skip for bug fixes, copy tweaks, small CRUD.
Notable: two-phase — Phase 1 architecture, Phase 2 flows + contracts + shared patterns. Treats existing code as source of truth, doesn't reinvent what's there.

Skill 03 / 09

ui-ux-design-generator

/ui-ux-design-generator

A design-system skill: UX wireframes first (how it works), then a UI design system spec (how it looks).

In

A design ask — new system, new feature, or modification of an existing component.

Out

UX: wireframe HTMLs + optional live demo + UX brief. UI: design-system spec (palette, type, motion, components) + optional icon set. All in specs/ui-ux-design-<name>/.

When to invoke: when you need interaction patterns validated before development; when picking a visual style for a new surface.
Notable: two-phase (UX → UI), each phase asks how many options to generate, and a state file lets you resume across sessions.

Artifacts & why two phases

What lands on disk

specs/ui-ux-design-reports/ ├── .design-state.json # resume state ├── ux-preview-option-a.html # wireframe, interactive ├── ux-preview-option-b.html # alternate wireframe ├── ux-live-demo.html # optional · full-flow simulation ├── ux-brief.md # UX patterns & decisions ├── ui-preview-option-a.html # design-system applied to UX ├── ui-preview-option-b.html ├── ui-brief.md # palette, type, motion, components └── icon-set.md # optional · custom SVG icons

Why two phases

Phase 1 — UX answers "is this the right flow?" Wireframes are interactive and grayscale on purpose: you validate behavior, navigation, and states without arguing about color. The optional live demo simulates the full end-to-end experience.

Phase 2 — UI answers "how does the system look?" The deliverable is a design-system spec — color tokens, type scale, motion, spacing, component styles. The HTML previews just visualize it; the brief is what frontend implements against.

State persists in .design-state.json, so any phase can be paused and resumed.

Skill 04 / 09

create-implementation-order

/create-implementation-order

Looks at all your specs and proposes a dependency-ordered implementation strategy — one plan or many.

In

All spec files: use cases, design, flows, shared patterns.

Out

specs/implementation-order.md — domain groupings with the right /dev-plan commands to invoke for each.

When to invoke: after design is approved, before you start implementing — when you need to decide "is this one plan, or three?".
Notable: small scope → one plan; large scope → multiple domain plans grouped by shared infra. Never invokes /dev-plan itself — that's your call.

Skill 05 / 09

dev-plan

/dev-plan

Break a feature into minimal, independently committable steps — each leaving the system working and reviewable.

In

A task description plus the codebase context.

Out

specs/plan-<name>/ with a README (status tracker) plus step-01-*.md, step-02-*.md, … each with goal, changes, acceptance criteria, code examples, tests.

When to invoke: before writing code on anything non-trivial — or to resume an existing plan tomorrow.
Notable: the README is the state machine. Future sessions read it to know what's done. Each step ships as its own commit.

Skill 06 / 09 · the orchestrator

end-to-end

/end-to-end

Drives the whole pipeline for you — requirements → use-cases → tech-design → ui-ux → impl-order → dev-plans → execution.

In

A new feature or system description + requirements.

Out

specs/e2e-<name>-<date>/ containing every downstream artifact: use-cases/, tech-design/, design/, plan-*/.

When to invoke: kicking off a new feature from scratch and you want the guided experience.
Notable: maintains workflow state JSON. At every transition: continue / go-back / stop. Does not auto-commit — that's still your decision.

Pipeline · the meta-skill
Skill 07 / 09

pre-commit

/pre-commit

Run the right quality gates — tests then build — before you commit. Stack-aware.

In

Just invoke it. Detects Node, Python, Go, Rust, etc. automatically.

Out

Pass / fail / skip per gate, with the relevant ~30 lines of error output if something breaks.

When to invoke: right before /commit, every time. Catches the silly stuff before review.
Notable: tests first (cheapest fix), build second. Stops on first failure. Never auto-commits — you decide after gates pass.

Pipeline · step 7
Skill 08 / 09

commit

/commit

Read the diff, write a clean Conventional Commit, stage the right files, create the commit.

In

Your current working tree (staged + unstaged changes).

Out

<type>[scope]: <description> commit with optional body and a Co-Authored-By footer.

When to invoke: after /pre-commit passes and you're ready to land a step.
Notable: stages files by name (never git add -A), refuses to stage anything that smells like a secret, never amends — always creates a new commit.

Skill 09 / 09

document

/document

Create or update docs to reflect what the code actually does. Code is the source of truth.

In

A topic ("authentication", "ingestion pipeline") — and optionally an existing doc to align.

Out

A markdown doc in docs/, registered in CLAUDE.md with trigger hints so future agents find it.

When to invoke: after a feature lands; whenever docs and code have drifted.
Notable: two modes — update an existing doc, or create new (asks doc type and audience first). If code and docs disagree, code wins.
